## L-1. 「古い」「未使用」の判定は事実確認を経てから

**発生**: 2026-04-19, F-1（Emacs古いパッケージ削除）作業中
**誤り**: `helm`, `auto-complete`, `smartparens`, `org2blog`, `enh-ruby-mode` を「古い」と判断して削除しようとした
**実態**: すべて 2025-2026 に更新のある現役パッケージ（GitHub API で確認）
**教訓**:
- 「古い」と言うなら **最終更新日 / archived フラグ** で裏取りする
- 「使っていない」と言うなら **ユーザーに実使用を確認** する
- 計画ドキュメントにある項目でも、削除前に事実確認を一段挟む
- 特に破壊的な削除（ファイル・パッケージ）は F-1 のような巻き戻しコストが高い

**適用場面**: Brewfile / パッケージリスト / 設定ファイルの削除系タスク全般

## L-2. `git rm` 済みファイルを `git add` しない

**発生**: B-1, D-4 での commit-draft 失敗
**誤り**: `git rm` で staging 済みの削除ファイルを、commit-draft スクリプトの `git add` に含めてしまい `pathspec did not match any files` エラー
**教訓**:
- commit-draft 生成前に `git status` で staged / unstaged を確認する
- すでに staged な削除は `git add` から外す。新規の modified のみ `git add` する

**適用場面**: `git rm` で削除したファイルを含むコミットを作るとき

## L-3. fish exec をしない理由は「fish 非対応・動作が不安定」

**発生**: 2026-04-19, C-2（`should_exec_fish` 関数化）で誤ったコメントを書いた
**誤り**: zshrc_mac で IDE / Warp で `exec fish` しない理由を「それぞれ独自の補完・履歴機能があるため」と記述した
**実態**: 真の理由は「fish に対応していない or fish 起動で動作が不安定になるツールがあるため個別判定で zsh に残している」
**教訓**:
- 既存コードの意図を勝手に解釈しない。条件の理由が書かれていないときは確認するか、事実ベース（「このツールでは fish を起動しない」）だけ書く
- IDE 独自機能の有無は、zsh に残す理由ではない。重要なのは **fish が動くかどうか**

**適用場面**: `zshrc_mac` / `zshrc_linux` の `should_exec_fish` 判定を拡張するとき。新しいツールを追加する際は「fish がそこで動くか・安定するか」で判断する
